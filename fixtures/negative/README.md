# Negative Fixtures (Reader Error-Path Corpus)

Generated by `scripts/generate-negative-fixtures.py` from a fixed source list
(currently four representative archives across the 1.5/3.x/4.x range). Each
source produces three deterministic corruption variants:

| Suffix | Corruption | Tests |
|---|---|---|
| `.trunc1.<ext>` | Truncated to 1 byte | Reader rejects on signature mismatch (file too short to contain even the marker). |
| `.trunc-half.<ext>` | Truncated to `len // 2` | Reader hits unexpected EOF mid-archive — ideally during header walk or compressed-data read. |
| `.bitflip.<ext>` | Bit `0x40` flipped at offset `min(32, len-1)` | Reader sees a structurally-plausible-looking byte change inside the first/main header. Should fail CRC on the affected block. |

`MANIFEST.tsv` lists every output with size + SHA-256 + provenance.

## Source archives

- `fixtures/1.54/readme_154_normal.rar` — RAR 1.5 single file (Unpack15)
- `fixtures/1.54/readme.EXE` — RAR 1.5 SFX (RAR data appended to MZ stub)
- `fixtures/1.54/random.rar` — RAR 1.5 multi-volume large file
- `fixtures/rarvm/archives/delta_4ch_rar393.rar` — RAR 3.x with VM filter

The set is intentionally small. Each new spec-text rule about reader-side
rejection (broken header, unexpected EOF, signature corruption, CRC failure)
should add a corresponding source to this list rather than producing
hand-crafted fixtures.

## Reader behaviour expected

A correct reader handling `.trunc1`:
- Reports "not a RAR archive" or equivalent (signature check fails).

A correct reader handling `.trunc-half`:
- Reads the marker and main header, then encounters EOF either mid-block
  header or mid-payload. Should report "unexpected end of file at block
  type X" per `READ_SIDE_OVERVIEW.md` §10.

A correct reader handling `.bitflip`:
- The bit-flip lands inside or near the main header (offset 32 puts it past
  the marker for both RAR 1.5 and RAR 5.0 framings). The affected block's
  CRC should fail; in lenient mode the reader skips to the next block via
  `HeadSize`, in strict mode it aborts.

These are coarse smoke tests, not exhaustive negative coverage. A future
fuzz-style corpus would exercise byte-flips at every offset for each block
type.
